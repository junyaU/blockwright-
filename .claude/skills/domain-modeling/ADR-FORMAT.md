# ADR の形式

ADR は `docs/adr/` に置き、連番で命名する: `0001-slug.md`, `0002-slug.md` ……。

`docs/adr/` ディレクトリは遅延作成する ── 最初の ADR が必要になったときだけ。

## テンプレート

```md
# {決定の短いタイトル}

{1〜3文: 文脈は何か、何を決めたか、なぜか。}
```

これだけ。ADR は1段落で構わない。価値は、決定が*なされた*ことと*その理由*を記録することにあり、セクションを埋めることにはない。

## 任意セクション

本当に価値を加えるときだけ含める。ほとんどの ADR には不要。

- **Status** フロントマター（`proposed | accepted | deprecated | superseded by ADR-NNNN`） ── 決定を見直す可能性があるときに有用
- **Considered Options（検討した選択肢）** ── 却下した代替案を覚えておく価値があるときだけ
- **Consequences（帰結）** ── 自明でない下流への影響を明示する必要があるときだけ

## 採番

`docs/adr/` を走査して最大の既存番号を見つけ、1 を足す。

## ADR を提案すべきとき

次の3つがすべて真でなければならない:

1. **後戻りが難しい** ── 後で考えを変えるコストが意味のある大きさ
2. **文脈なしでは意外** ── 将来の読者がコードを見て「いったいなぜこうしたのか？」と訝る
3. **本物のトレードオフの結果** ── 真の選択肢があり、特定の理由で1つを選んだ

後戻りが容易な決定は見送る ── どうせ覆すだけだ。意外でないなら、誰も理由を訝らない。真の代替案が無かったなら、「自明なことをした」以上に記録すべきものはない。

### 該当するもの

- **アーキテクチャの形。**「モノレポを使う。」「書き込みモデルはイベントソーシング、読み取りモデルは Postgres へ射影する。」
- **コンテキスト間の統合パターン。**「Ordering と Billing は同期 HTTP ではなくドメインイベントで通信する。」
- **ロックインを伴う技術選定。** データベース、メッセージバス、認証プロバイダ、デプロイ先。すべてのライブラリではなく、差し替えに四半期かかるようなものだけ。
- **境界とスコープの決定。**「顧客データは Customer コンテキストが所有し、他のコンテキストは ID でのみ参照する。」明示的な「しない」は「する」と同じくらい価値がある。
- **自明な道からの意図的な逸脱。**「X の理由で ORM ではなく手書き SQL を使う。」まともな読者なら逆を想定するようなもの全般。これは次のエンジニアが「意図的だったもの」を『修正』するのを防ぐ。
- **コードに見えない制約。**「コンプライアンス要件のため AWS は使えない。」「パートナー API の契約上、応答時間は 200ms 未満でなければならない。」
- **却下が自明でない場合の却下した代替案。** GraphQL を検討して微妙な理由で REST を選んだなら記録する ── さもないと半年後に誰かがまた GraphQL を提案する。
