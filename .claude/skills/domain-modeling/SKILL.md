# ドメインモデリング

設計を進めながら、プロジェクトのドメインモデルを能動的に構築・研磨する。これは*能動的*な規律である ── 用語に異議を唱え、エッジケースのシナリオを考案し、結晶化した瞬間にグロッサリと決定を書き留める。（語彙のために `CONTEXT.md` を*読むだけ*はこのスキルではない。それはどのスキルでもできる一行の習慣にすぎない。このスキルは、モデルを消費するのではなく*変更する*ときのためのものだ。）

## ファイル構成

ほとんどのリポジトリはコンテキストが1つ:

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

ルートに `CONTEXT-MAP.md` があれば、そのリポジトリは複数コンテキストを持つ。マップが各コンテキストの所在を指す:

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← システム全体の決定
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← コンテキスト固有の決定
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

ファイルは遅延作成する ── 書くべきものができたときだけ。`CONTEXT.md` が無ければ、最初の用語が確定したときに作る。`docs/adr/` が無ければ、最初の ADR が必要になったときに作る。

## セッション中

### グロッサリと突き合わせて異議を唱える

ユーザーが `CONTEXT.md` の既存語彙と矛盾する用語を使ったら、即座に指摘する。「グロッサリでは『キャンセル』を X と定義していますが、あなたは Y を意図しているように見えます ── どちらですか？」

### 曖昧な言葉を研ぎ澄ます

ユーザーが曖昧・多義的な用語を使ったら、精密な正準用語を提案する。「『アカウント』と言っていますが、Customer のことですか、それとも User のことですか？ それらは別物です。」

### 具体的なシナリオで議論する

ドメインの関係が議論されているときは、具体的なシナリオでストレステストする。エッジケースを突くシナリオを考案し、概念どうしの境界についてユーザーに精密さを強いる。

### コードと相互参照する

ユーザーが「こう動く」と述べたら、コードがそれに同意するか確認する。矛盾を見つけたら表面化させる。「あなたのコードは Order 全体をキャンセルしていますが、いま部分キャンセルが可能と言いました ── どちらが正しいですか？」

### CONTEXT.md をその場で更新する

用語が確定したら、その場で `CONTEXT.md` を更新する。まとめてやらず、発生した瞬間に捕える。形式は [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) に従う。

`CONTEXT.md` には実装の詳細を一切含めない。`CONTEXT.md` を仕様書・スクラッチパッド・実装決定の置き場として扱わないこと。それはグロッサリであり、それ以外の何物でもない。

### ADR は控えめに提案する

次の3つがすべて真のときだけ ADR の作成を提案する:

1. **後戻りが難しい** ── 後で考えを変えるコストが意味のある大きさ
2. **文脈なしでは意外** ── 将来の読者が「なぜこうしたのか？」と訝る
3. **本物のトレードオフの結果** ── 真の選択肢があり、特定の理由で1つを選んだ

3つのどれかが欠けるなら、ADR は見送る。形式は [ADR-FORMAT.md](./ADR-FORMAT.md) に従う。
