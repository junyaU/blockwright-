# CONTEXT.md の形式

## 構造

```md
# {コンテキスト名}

{このコンテキストが何であり、なぜ存在するのかを1〜2文で説明}

## 用語

**Order（注文）**:
{用語の説明を1〜2文で}
_避ける_: Purchase, transaction

**Invoice（請求書）**:
納品後に顧客へ送られる支払い要求。
_避ける_: Bill, payment request

**Customer（顧客）**:
注文を行う個人または組織。
_避ける_: Client, buyer, account
```

## ルール

- **断定的であること。** 同じ概念に複数の語があるなら、最良の1つを選び、残りは `_避ける_` に並べる。
- **定義は引き締めること。** 最大1〜2文。それが「何をするか」ではなく「何であるか」を定義する。
- **このプロジェクトのコンテキストに固有な用語だけを含めること。** 一般的なプログラミング概念（タイムアウト、エラー型、ユーティリティのパターン）は、プロジェクトで多用されていても含めない。用語を加える前に問う ── これはこのコンテキスト固有の概念か、それとも一般的なプログラミング概念か？ 前者だけが含まれる。
- **自然なまとまりが生まれたら、用語を小見出しでグループ化すること。** すべての用語が単一のまとまった領域に属するなら、フラットなリストで構わない。

## 単一コンテキスト vs 複数コンテキストのリポジトリ

**単一コンテキスト（ほとんどのリポジトリ）:** リポジトリルートに `CONTEXT.md` が1つ。

**複数コンテキスト:** リポジトリルートの `CONTEXT-MAP.md` が、各コンテキストとその所在、相互の関係を列挙する:

```md
# Context Map

## Contexts

- [Ordering](./src/ordering/CONTEXT.md) — 顧客注文の受付と追跡
- [Billing](./src/billing/CONTEXT.md) — 請求書の生成と支払い処理
- [Fulfillment](./src/fulfillment/CONTEXT.md) — 倉庫のピッキングと出荷の管理

## Relationships

- **Ordering → Fulfillment**: Ordering が `OrderPlaced` イベントを発行し、Fulfillment がそれを消費してピッキングを開始する
- **Fulfillment → Billing**: Fulfillment が `ShipmentDispatched` イベントを発行し、Billing がそれを消費して請求書を生成する
- **Ordering ↔ Billing**: `CustomerId` と `Money` の型を共有する
```

スキルはどちらの構造が当てはまるかを推測する:

- `CONTEXT-MAP.md` があれば、それを読んでコンテキストを見つける
- ルートの `CONTEXT.md` だけがあれば、単一コンテキスト
- どちらも無ければ、最初の用語が確定したときにルートの `CONTEXT.md` を遅延作成する

複数コンテキストが存在する場合、現在の話題がどれに関係するかを推測する。不明なら尋ねる。
