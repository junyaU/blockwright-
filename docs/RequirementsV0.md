## 0. このドキュメントの読み方（Claude Codeへの前提指示）

- これは **v0（歩く骨格）** の要件である。**スコープを勝手に広げないこと。**「§4.2 やらないこと」に挙げた項目は、たとえ簡単に見えても v0 では実装しない。
- ただし **唯一の例外が「IR seam（§5・§6.3）」**。ここはミニマムでも構造だけは将来拡張に耐える形で必ず固定する。後から作り直しになる継ぎ目はここだけなので、ここの契約は厳守する。
- 不確実性（特に WebSocketプロトコルの詳細・座標解決）がある箇所は §11 に明記した。**推測で固定実装せず、まず疎通実験で挙動を確認してから実装すること**（このプロジェクトの非公式API依存ゆえの方針）。
- 設計思想は「動かしながら育てる」。**まず1本ループを通すことを最優先**し、品質・表現力は後続で足す。

## 1. 背景と目的

ゲーム内チャットで「家を建てて」のように話しかけると、AI（Claude）がその意図を構造化し、コードがそれを Minecraft 統合版の世界に実際のブロックとして組み上げる。

v0 のゴールは見た目の豪華さではなく、**入力からブロック設置までのループを一度完全に通すこと**。最小の成果物は「チャットで指示 → 目の前に箱が建つ → 取り消せる」。

## 2. 全体アーキテクチャ

### 2.1 データフロー

```
[1] 入力        [2] AI            [3] IR          [4] build()        [5] 設置
チャット発言 ──▶ Claude API ──▶ 中間表現(JSON) ──▶ コマンド変換 ──▶ Minecraftへ送信
(PlayerMessage)  意図を構造化     ★契約境界★        座標計算/施工      (WebSocket)
```

- `[3] IR` が **このシステムの背骨**。AI側（賢さ）とコード側（正確さ）を分離する契約境界（seam）。
- `[4] build()` 以降は **決定論的**であること。座標計算・素材解決の正しさはコードが保証し、AIには絶対に委ねない。
- `[2]` だけが外向き通信（Claude API はインターネット越し）。`[1][5]` のゲーム⇔サーバー間はローカル（WSL ⇔ Windows）。

### 2.2 コンポーネント

| # | コンポーネント | 責務 | 該当節 |
|---|---|---|---|
| C1 | WebSocketサーバー | MinecraftとのWS接続・イベント購読・コマンド送信 | §6.1 |
| C2 | Claude APIクライアント | 発言→IR(JSON)の生成、出力のパース・検証 | §6.2 |
| C3 | ビルダー `build()` | IR→コマンド列への変換、原点解決、素材検証 | §6.3 |
| C4 | Undoマネージャ | 設置領域の記録と取り消し | §6.4 |

### 2.3 最重要原則：seamの固定

```ts
function build(ir: IR, origin: Vec3): BuildResult
```

- **この署名は v0 以降も変えない。** IRの表現力を上げるときは `build()` の中身（type分岐）だけを差し替える。呼び出し側・原点解決・Undo・送信はノータッチで育てられる。
- 逆に、AIに生 `setblock` 文字列を吐かせて中継する実装は **禁止**（施工エンジンが存在しない＝後から全書き直しになるため）。

## 3. 実行環境

### 3.1 構成

- **Node.js**：WSL2（Ubuntu系）上で実行。LTS（v20以上）想定。
- **Minecraft**：Windows側の統合版（Bedrock, UWPアプリ）。
- したがって **Windows上のMinecraftが、WSL2上のWSサーバーへ接続する**構成になる。ここがネットワーク的に最大のハマりどころ（§3.2・§11）。

### 3.2 ネットワーク要件（人手で行う前提作業／コード対象外）

以下は実装コードではなく **環境セットアップ手順**。READMEに手順として記載すること。

1. **WSサーバーは `0.0.0.0` にバインドする**（`127.0.0.1` 固定にしない）。WSL2上のサーバーをWindows側から到達可能にするため。→ これは §7 FR-02 としてコード要件にも含む。
2. **WSL2のネットワーク到達性**を確保する。いずれかの方法：
   - 推奨：WSL2の **mirrored networking** を有効化（`%UserProfile%\.wslconfig` に `[wsl2]\nnetworkingMode=mirrored`）。これで Windows から `localhost:<port>` で WSL のサーバーに届く。
   - 代替：WSL の IP（`hostname -I` で確認）を使い、ゲーム内で `/connect <WSLのIP>:<port>` する。
3. **ループバック制限の解除**（localhost/ループバック経由で繋ぐ場合に必要）。Windowsの管理者PowerShellで一度だけ実行：
   ```powershell
   CheckNetIsolation LoopbackExempt -a -n="Microsoft.MinecraftUWP_8wekyb3d8bbwe"
   ```
4. **ファイアウォール**：使用ポートへの受信を許可（WSL IP直結方式の場合に特に）。

> セットアップが正しいかは「サーバー起動 →（チート有効ワールドで）`/connect <host>:<port>` → 接続成功メッセージが出る」で確認する。無言で失敗する場合はほぼ上記1〜4のどれか。

### 3.3 設定値

- ポート番号：設定可能にする（既定例 19131）。環境変数 or 設定ファイル。
- `ANTHROPIC_API_KEY`：環境変数（`.env`）で受け取り、リポジトリにコミットしない。

## 4. スコープ

### 4.1 v0でやること（In Scope）

- WSサーバーを立て、Minecraftからの `/connect` を受け付ける。
- `PlayerMessage` イベントを購読する。
- 建築を指示する発言（トリガー語）を検知したら Claude API を呼ぶ。
- Claude に **箱(box)型のIR** を生成させる。
- `build(ir, origin)` でIRを `fill` コマンドに変換し、**プレイヤーの現在位置を基準（原点）**に設置する。
- 直前の建築を **領域単位でUndo** できる。
- 不正な素材IDが来ても落ちずにフォールバックする。
- 基本的なロギングを行う（§8）。

### 4.2 v0でやらないこと（Out of Scope：明示的に実装しない）

- グリッドIR、壁/ドア/窓/屋根などの構造表現（boxのみ）。
- 画像・設計図入力。
- 建物タイプの複数化（家・塔・橋など）。
- 素材スタイル/プリセットの語彙体系。
- ブロック単位の完全なUndo（領域単位のair埋め戻しのみ）。
- マルチプレイヤー対応・複数同時建築・キューイング。
- 永続化（DB等）。状態はプロセスメモリで十分。

### 4.3 将来の拡張軸（設計上の余地は残すが、v0では実装しない）

各軸が「どの層に付くか」を意識した設計にしておくこと（コードコメント程度で良い）。

| 拡張軸 | 付く層 | 設計上の含意 |
|---|---|---|
| IR表現力（box→grid→house） | seam両側（§5の`type`分岐） | IRに `type` を持たせ、`build()` を type で分岐させておく |
| 素材・スタイル語彙 | 主にAI側＋検証テーブル | 素材は意味スロット（palette）として持つ余地を残す |
| 画像/設計図入力 | IRの手前に新ステージ | 入力フロントを差し替え可能に。出口がIRに揃えば下流は不変 |
| 建物タイプ複数化 | AI側のレパートリー | IRが表現力を持てばプロンプト追加のみで増える |
| Undo（完全版） | seamの裏（`build()`下） | ブロック設置を必ず1箇所のチョークポイントに通す |

## 5. 中間表現（IR）仕様 ★最重要★

### 5.1 設計方針

- IRは **`type` で判別される判別可能ユニオン（discriminated union）** とする。v0では `box` のみだが、将来 `grid`・`house` を**追加できる形**で定義する。
- IRは **絶対座標を持たない**。サイズ・素材など「何を建てるか」だけを表し、「どこに建てるか（原点）」は `build()` の引数として外から与える。これにより同じIRを任意の場所に建てられる。

### 5.2 v0 スキーマ（TypeScript型で精密に定義）

```ts
type Vec3 = { x: number; y: number; z: number };

// v0 では Box のみ。将来 BoxIR | GridIR | HouseIR のユニオンに拡張する。
type IR = BoxIR;

interface BoxIR {
  type: "box";                       // discriminator（将来の分岐キー）
  size: { w: number; d: number; h: number }; // ブロック数。各 1..64 を許容範囲とする
  material: string;                  // BEブロックID（例 "minecraft:oak_planks"）。検証対象
  hollow?: boolean;                  // 中空にするか。省略時 false（中身も詰まった箱）
}

// build() の返り値
interface BuildResult {
  region: { min: Vec3; max: Vec3 };  // 設置領域の絶対座標（Undo用）
  commands: string[];                // 送信したコマンド列（ログ/デバッグ用）
}
```

### 5.3 制約・バリデーション

- `size.w/d/h` は整数、各 `1..64`。範囲外はクランプ（丸め）するか拒否する（v0はクランプで可）。`fill` の体積上限（後述）にも留意。
- `material` は **§6.3の素材検証**を必ず通す。検証前のIRをそのまま施工に渡さない。
- IRはClaudeの出力をパースして得る。**パース失敗・スキーマ不一致時は施工せず、ユーザーにゲーム内チャットで失敗を通知する**（無言で失敗しない）。

## 6. コンポーネント詳細仕様

### 6.1 C1: WebSocketサーバー

- 役割：Minecraft（クライアント側）からの接続を待ち受けるWSサーバーとして動作する。Minecraftが `/connect` でこのサーバーに繋いでくる。
- バインド：`0.0.0.0:<port>`（§3.2-1）。
- 接続確立後、**イベント購読メッセージを送る**：`PlayerMessage` を購読する。
- 受信した `PlayerMessage` イベントを上位（トリガー判定→C2）に渡す。
- コマンド送信メッセージを組み立てて送る機能を提供する（C3が生成したコマンド列を順次送信）。
- 各メッセージの `header.requestId` には **UUIDv4** を都度生成して付与する。

> プロトコルの想定メッセージ形は §付録A に記す。**version差分で変わりうるため、まず購読が効いて発言イベントが届くか・コマンドが実際に通るかを疎通実験で確認してから本実装に進む**こと。

### 6.2 C2: Claude APIクライアント

- 役割：ユーザー発言を入力に、**IR(JSON)のみ**を出力させる。
- モデル：既定は `claude-sonnet-4-6`（速度・コスト重視なら haiku 系も可）。モデル文字列は設定で差し替え可能にし、最新の利用可能モデルを確認すること。
- システムプロンプト要件：
  - 出力は **§5.2のスキーマに厳密準拠したJSONのみ**。前置き・後置き・Markdownコードフェンス（```）を**一切付けない**よう明示。
  - v0では必ず `type: "box"` を返すよう指示。
  - 不明な素材は **存在するBEブロックID**で返すよう促す（ただし最終防衛は§6.3のコード側検証）。
- 出力ハンドリング：
  - レスポンスから本文テキストを取り出し、**前後の余分（フェンス等）を除去してからJSON.parse**する。
  - パース成功後、スキーマ検証（§5.3）。失敗時はリトライ1回 → なお失敗ならユーザーに失敗通知。
- APIキーは環境変数。SDK（`@anthropic-ai/sdk`）または `fetch` 直叩きのいずれでも可。

### 6.3 C3: ビルダー `build(ir, origin)`

- 署名：`build(ir: IR, origin: Vec3): BuildResult`（**不変契約**）。
- **原点解決**：`origin` は「プレイヤーの現在の絶対座標」を基準に与える。
  - ⚠ WebSocket経由で実行するコマンドは相対座標 `~ ~ ~` がプレイヤー足元に素直に解決しない場合がある（§11）。そのため **プレイヤー座標を一度問い合わせて絶対座標を得てから**施工する。問い合わせ方法（例：`querytarget` 系コマンドの応答をパース、または `execute` でのラップ）は疎通実験で確定する。
  - 箱はプレイヤーの少し前方/足元基準など、自然な位置に置く（具体オフセットはv0では固定値で良い）。
- **IR→コマンド変換（box）**：
  - `type==="box"` のとき、`size` と `origin` から領域の `min`/`max` 絶対座標を算出。
  - `fill <min> <max> <material>` を生成。`hollow:true` なら `fill ... hollow` を用いる。
  - **`fill` の体積上限に注意**（1コマンドで扱える上限を超える場合は領域を分割して複数 `fill` に。v0のsize上限64³前提でも上限を超え得るので分割ロジックを入れるか、上限内にsizeをクランプする）。
- **素材検証**：
  - 既知のBEブロックID集合（v0は小さなallowリストで可）と照合。
  - よくある **JE名→BE名のエイリアス補正**（例の方針：`grass`→`grass_block` 等、必要分のみ）。
  - 不明IDは **フォールバック**（例 `minecraft:stone`）に置換し、ログに警告。**ここで例外を投げて全体を止めない。**
- 返り値 `BuildResult` に設置領域（`region`）と送信コマンド（`commands`）を入れて返す。`region` はUndo（C4）に渡す。

### 6.4 C4: Undoマネージャ

- 直近の `BuildResult.region` を保持する（v0は1段でよい。スタックにすれば複数段Undoに拡張可）。
- Undoトリガー語（例「もどして」「undo」）を検知したら、保持中の領域を **`fill <min> <max> minecraft:air`** で埋め戻す。
- ⚠ これは **領域単位Undo**であり、建築前の元状態を完全復元するものではない（地形・既存建築は戻らない）。v0の用途（同じ場所で建て直してテスト）には十分。この制約をREADMEに明記。
- 将来の完全版（structure block / `.mcstructure` でのスナップショット復元）への置換余地を、`build()`の裏のチョークポイントとして残しておく。

## 8. 非機能要件

- **ロギング（必須）**：WS非公式APIゆえデバッグが要になる。最低限、(a)送受信した全WSメッセージ、(b)生成IR、(c)送信コマンド、(d)素材フォールバック等の警告、をログ出力する。レベル分け（info/warn/error）推奨。
- **エラーハンドリング**：いかなる単一の失敗（API失敗・パース失敗・コマンド失敗・WS切断）でも **プロセス全体を落とさない**。可能ならユーザーにゲーム内チャットでフィードバックする。
- **設定の外出し**：ポート、APIキー、モデル名、トリガー語、フォールバック素材を設定値として扱う。
- **再接続**：WS切断時に致命終了せず、再接続できると望ましい（v0では手動再起動許容でも可）。
- **秘匿情報**：APIキーをログ・リポジトリに出さない。`.env` を `.gitignore`。

## 付録A：WSメッセージ想定形（※要・疎通確認）

> 下記は一般に知られる形。**version依存で変わりうるため、実装前に実際のやり取りで検証すること。**

イベント購読（サーバー→Minecraft）：

```json
{
  "header": {
    "version": 1,
    "requestId": "<UUIDv4>",
    "messageType": "commandRequest",
    "messagePurpose": "subscribe"
  },
  "body": { "eventName": "PlayerMessage" }
}
```

コマンド実行（サーバー→Minecraft）：

```json
{
  "header": {
    "version": 1,
    "requestId": "<UUIDv4>",
    "messageType": "commandRequest",
    "messagePurpose": "commandRequest"
  },
  "body": {
    "version": 1,
    "commandLine": "fill 10 64 10 14 68 14 minecraft:oak_planks",
    "origin": { "type": "player" }
  }
}
```

`PlayerMessage` イベント（Minecraft→サーバー）の `body` には発言者・本文・種別などが含まれる。トリガー判定と座標問い合わせの応答パースは、この実データを見て確定する。

## 付録B：v0以降の最初の一歩（参考）

seam が固定されていれば、次の一手は `build()` 内の分岐追加から始まる：

```ts
function build(ir: IR, origin: Vec3): BuildResult {
  switch (ir.type) {
    case "box":  return buildBox(ir, origin);   // v0
    // case "grid": return buildGrid(ir, origin); // 将来：層ごと2Dグリッド
    // case "house":return buildHouse(ir, origin);// 将来：壁/ドア/窓/屋根
