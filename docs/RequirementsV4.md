## 0. このドキュメントの読み方（Claude Codeへの前提指示）★重要な方針転換あり★

- 本書は **v0〜v3 への差分**。`build(ir, origin)` 署名不変・IRは絶対座標を持たない・buildGrid以降の不変、は継続。
- **⚠ 本版は設計原則を意図的に1か所だけ破る。** v0〜v3は「**AIに形（幾何）を作らせない＝決定論ボクセル化**」で一貫していた。v4は **stage3（image→3D生成）でのみ、AI/生成モデルに形を作らせる**。これは「事前データ無しで喋るだけで立体を建てたい」という要求を満たすための、**自覚的なトレードオフ**である。
- **したがって v4 設計の核心は「原則破りの封じ込め」**：
  - 生成（形を作る）は **stage3 だけ**に閉じ込める。他段に生成を漏らさない（§2.1）。
  - 生成の荒さは **stage4（cleanup）で吸収**し、**stage5以降（voxelize/build）の決定論を死守**する（§2.2）。
  - 破綻したら **v3.0 平面建築にフォールバック**して、最低限の成果物を必ず返す（§2.3）。
- **個人・ローカル利用前提**（§3・§5.2・R3）。版権キャラの取得・3D化を含むため、再配布・公開・サービス化はスコープ外。
- スコープを広げない。特に「text→3D（画像を介さない生成）」「高忠実の追求」はやらない（§5.2）。

## 1. 背景と目的

v3 までで「リファレンス（画像/3Dモデル）があれば建つ」ところまで来た。だが立体キャラには **事前に3Dモデルを用意する**必要が残っていた。v4 のゴールは、**「ロボット作って」と言うだけで、事前データ無しに立体が建つ**こと。多少荒くてよい（忠実さより手間ゼロを優先）。

そのために、裏側で「意図抽出 → 画像取得 → image→3D生成 → 正規化 → ボクセル化 → 建築」を自動で回す。④以降は v3 の資産をそのまま使い、新規は前段（①〜④）に集約する。

---

## 2. 設計思想

### 2.1 原則破りを stage3 に封じ込める

```
① 意図抽出    "robot" を特定（AI＝言語処理、形に触れない）
② 画像取得    ネットから参照画像1枚（外部）
③ image→3D    外部生成サービスで .glb 化  ★ここだけAIが形を作る（原則破り）★
④ 正規化/清掃  生成メッシュをクリーンアップ（荒さの吸収層）
⑤ ボクセル化  v3.1 エンジンで GridIR 化（決定論・不変）
⑥ 建築        build() → ゲームに建つ（不変）
```

- **形を生成するのは ③ のみ。** ①は言語、②は取得、④は決定論クリーンアップ、⑤⑥は決定論。生成の不確実性を1段に局所化する。
- これにより「AIが形に触れる範囲」を最小化し、v3までの決定論パイプラインを最大限温存する。

### 2.2 荒さを吸収する cleanup 層（v4の肝）

- 生成3Dは**スケール・向き・浮遊片・穴**がバラバラ。これを生のまま⑤に流すと、ボクセル化がゴミを拾って原型を留めない。
- **③と⑤の間に正規化/清掃（④）を必ず噛ませる**。再センタリング・スケール正規化・向き補正・**最大連結成分以外の破片除去**を行い、「⑤が安心して食えるメッシュ」に整える。
- 設計の合言葉：**荒さは③で生まれ、④で吸収し、⑤以降には漏らさない。**

### 2.3 フォールバックを最後の砦にする

- ②③④⑤のどこかで破綻したら、**②で取得済みの画像を使って v3.0 平面建築にフォールバック**する。立体が無理でも「板ロボット」を必ず返す。
- フォールバック自体は**決定論（v3.0）で堅い**。生成の不確実性に対する保険として機能する。
- 破綻検知は**品質ゲート**（⑤後の占有チェック等、§6.5）で行う。

## 5. スコープ

### 5.1 v4でやること（In Scope）

- **①意図抽出**：チャット発話から対象（"robot"等）と任意のサイズ指定を抽出（AI＝言語）。
- **②画像取得**：外部から参照画像を1枚取得（なるべく正面・単体・無地背景を優先）。フォールバック用に保持。
- **③image→3D生成**：外部APIへ画像を投入し .glb 等を取得（非同期ポーリング対応）。**唯一の生成（原則破り）ステップ**。
- **④正規化/清掃**：再センタリング・スケール正規化・向き補正・最大連結成分抽出（破片除去）・穴の軽処理。
- **⑤ボクセル化**：v3.1 を不変利用（メッシュ→GridIR）。
- **⑥建築**：build() を不変利用。
- **品質ゲート＆フォールバック**：破綻検知 → v3.0 平面建築（②の画像を使用）。
- **任意の背景除去**（③前処理、生成品質向上）。

### 5.2 v4でやらないこと（Out of Scope：明示）

- **生成物の外部公開・再配布・サービス化**。**個人ローカル利用限定**（版権キャラを含むため、§R3）。
- **画像を介さない text→3D 生成**（②を飛ばしてAIに丸ごと作らせる）。荒さの基準すら無くなるためやらない。
- **高忠実の追求**。v4は「荒くてよい」前提。精度を上げるのは別軸。
- **⑤⑥（voxelize/build）の再実装**。不変。
- **向き補正の完全自動化**（生成メッシュのforward軸特定は限界あり、§R2）。ベストエフォート。
- 複数キャラ同時・アニメーション・ポーズ指定。

### 5.3 将来拡張（設計余地）

- 画像取得元・3D生成サービスを差し替え可能に（アダプタ化）。
- 向き補正・破片除去を強化（生成品質に追従）。
- フォールバックを多段化（立体失敗→簡易立体→平面）。

## 6. コンポーネント詳細

### 6.1 ①意図抽出（AI・言語のみ）

- 入力：チャット発話（例「ロボット作って」「大きいロボット」）。
- 出力：`{ subject: string, targetHeight?: number }`。subjectは検索/生成に使う語（例 "robot"）。
- **AIは言語処理のみ**。形・座標・voxelには一切関与しない。
- 不明・曖昧な対象はユーザーに確認 or 失敗通知（無言失敗禁止）。

### 6.2 ②画像取得（外部）

- subject で画像検索し、**1枚を選定**。なるべく **正面・単体・無地背景・大きめ**の画像を優先（③の品質を左右するため）。
- 取得画像は **フォールバック用に保持**（§2.3）。
- 取得失敗（候補なし/通信失敗）→ 立体も平面も不可なので**失敗通知**して終了。
- 任意：**背景除去**で被写体を切り出し（③の生成品質向上）。

### 6.3 ③image→3D生成（外部・★原則破り★）

- 取得画像を外部生成サービスへ投入し、メッシュ（.glb/.obj）を得る。
- **非同期ジョブ**前提：投入→ポーリング→完了後DL。**タイムアウト・リトライ**を持つ（§R5）。
- 取得メッシュをローカル（`assets/generated/`等）に保存し、④へ渡す。
- 失敗/タイムアウト → **フォールバック分岐へ**（§6.6）。
- ⚠ **生成（形を作る）のはこの段のみ。** 他段に生成ロジックを置かない。

### 6.4 ④正規化/清掃（荒さ吸収層・決定論）

生成メッシュを⑤が安全に食える状態へ整える。すべて決定論：

- **再センタリング**：AABB中心を原点へ。
- **スケール正規化**：targetHeight（無ければ既定）に合わせ、v2.xサイズ上限（≤64）に収める。
- **向き補正**：被写体が正立・正面を向くようベストエフォートで補正（完全自動は不可、§R2）。最低限「上下が逆さま」を避ける。
- **最大連結成分の抽出**：浮遊した小片を捨て、主要部分のみ残す（生成ゴミ除去）。
- 任意：軽度の穴埋め・非多様体の整理。
- 出力：清掃済みメッシュ。

### 6.5 ⑤ボクセル化＋品質ゲート

- ⑤本体は **v3.1 を不変利用**（メッシュ→GridIR、shell/solid、表面色サンプリング、Lab色量子化、realized palette）。
- **品質ゲート（破綻検知）**：⑤の結果 GridIR に対し以下を判定：
  - 占有voxelが極端に少ない（ほぼ空）／極端に多い（ほぼ全solid＝塊化）。
  - AABBが退化（薄すぎ・潰れ）。
  - 最大連結成分が全体の妥当割合を下回る（破片だらけ）。
- いずれか該当 → **破綻とみなしフォールバック**（§6.6）。閾値はログを見て調整可能に（§R8）。

### 6.6 フォールバック（v3.0平面・決定論）

- 発動条件：②取得後の③④⑤のいずれかで失敗、または⑤の品質ゲート不合格。
- 処理：**②で取得済みの画像**を v3.0（平面画像→GridIR）に通し、板状で建てる。
- フォールバック自体は決定論で堅く実装。**これも失敗したら明示通知**（多重に黙らない）。
- ユーザーには「立体生成に失敗したため平面で建てた」旨を一言通知する。

### 6.7 ⑥建築・Undo連携

- 立体経路・平面フォールバックのいずれも、最終的に GridIR を `build(ir, origin)` に渡すだけ。**run-merge・facing・体積分割・全体AABB Undo は v2.x/v3 のまま**。再実装しない。

- ファイル構成例：

## 付録A：パイプライン擬似コード

```ts
async function makeFromUtterance(utterance: string, origin: Vec3) {
  const intent = await extractIntent(utterance);          // ① AI言語
  const image  = await acquireImage(intent.subject);      // ② 外部・保持
  if (!image) return notifyFail("画像が見つかりませんでした");

  try {
    const mesh   = await generate3D(image);               // ③ 外部・原則破り
    const clean  = cleanupMesh(mesh, intent.targetHeight);// ④ 決定論
    const ir     = voxelizeMesh(clean, "solid");          // ⑤ v3.1 不変
    if (!qualityGate(ir)) throw new Error("degenerate");  //   破綻検知
    return build(ir, origin);                             // ⑥ 立体で建つ
  } catch (e) {
    log.warn("立体生成に失敗、平面へフォールバック", e);
    notify("立体生成に失敗したため平面で建てます");
    const ir = voxelizeImageFlat(image);                  // v3.0 決定論
    return build(ir, origin);                             // 板で建つ
  }
}
```

形を生成するのは `generate3D`（③）のみ。`cleanup`・`voxelize`・`build`・`fallback` はすべて決定論。

## 付録B：ロードマップ上の位置づけ（総括）

```
v0  箱（ループ＋seam）           — 動かす土台
v1  house（パラメトリック）       — AI=設計, コード=施工
v2  タイプ複数化（生成器ライブラリ）— AI=選択+穴埋め
v2.x grid（自由形状の器・密版）    — 器を先に固める
v3  ボクセル化（充填手段）         — リファレンス→忠実に建てる
v4  喋るだけ（生成+取得+清掃+保険）— ★決定論原則をstage3でのみ破る★
```
