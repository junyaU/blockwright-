# 要件定義書：Minecraft統合版 AI建築システム v6

> 対象実装者：Claude Code
> 版：v6（経路ルーター＋リファレンス識別 ＝ 固有とジェネリックの振り分け）／前提：v0〜v5 実装済み
> 言語/環境：Node.js on WSL2、Minecraft Bedrock（Windows）
> 関連：`RequirementsV0〜V5.md` ／ `DESIGN_PRINCIPLES.md`（**改訂後の原則6**に準拠）

---

## 0. このドキュメントの読み方（Claude Codeへの前提指示）
- 本書は **v0〜v5 への差分**。`build(ir, origin)` 署名不変・IRは絶対座標を持たない・下流不変、は継続。
- **本書は改訂後の設計方針に立脚する**：本丸は「**喋るだけで固有の対象（特定の実在建造物・特定キャラ等）を建てる**」。原則6は「**対象の性質で経路を選ぶ（固有の対象は生成が本道、ジェネリックはパラメトリック）**」に改訂済み。
- **v6が解く課題**：v4は「新規建築をすべて生成に流す」前提だった。だが「東京タワー」も「塔」も同じ生成（or 同じ箱）になり、**固有の対象を作り分けられない / ジェネリックに無用な非決定論を使う**という不整合があった。v6は新規建築の手前に**分類と振り分け**を挟み、これを解消する。
- **v6の二本柱**：
  1. **経路ルーター**：発話を 固有／ジェネリック／曖昧 に分類し、適切な生成元へ振り分ける。
  2. **リファレンス識別**：固有名から**正しい参照**を取り違えずに同定する。生成エンジンの精度より、この「対象を間違えない」精度の方が固有再現に効く。
- **AIは言語と vision 判定（画像の一致確認）のみ。形・座標・voxelに触れない**（継続）。生成（形）は v4 stage3 だけ。
- スコープを広げない（§5.2）。

---

## 1. 背景と目的
v5までで「喋る→建てる→貯める→直す」は揃ったが、**「何を、どの経路で作るか」の判断が無かった**。その結果：
- 「東京タワー」と「スカイツリー」が**同じものになる**（固有の作り分け不能）。
- 「塔」のようなジェネリックな対象まで生成に流れ、**無用な非決定論**を被る。
v6のゴールは、**1つの入口（喋るだけ）を保ったまま、対象の性質に応じて最適な経路へ自動で振り分ける**こと。そして固有の対象では、**正しい参照を取ってくる精度**を中核に据える。

---

## 2. 設計思想
### 2.1 入口は1つ、経路は対象で分岐
```
発話 →（v5: 編集/削除/キャッシュ判定）→ 新規建築なら → ★v6 ルーター★
                                                          ├ ジェネリック → パラメトリック(v1/v2) → IR
                                                          ├ 固有       → リファレンス識別 → 生成(v4) → GridIR
                                                          └ 曖昧       → 既定ポリシー（§6.4）
                                                                                   ↓
                                                                       いずれも build(ir, origin) へ収束
```
- ユーザー体験は「喋るだけ」のまま。house/tower 等のタイプ名は表に出さない。
- 経路の違いは「**どの生成元が IR を用意するか**」の違いにすぎない。出口は常に IR → build()。下流不変。

### 2.2 誤分類の非対称性を意識する
- **固有 → ジェネリックの誤り**＝本丸の破壊（東京タワーがただの箱になる）。**最も避けるべき。**
- **ジェネリック → 固有の誤り**＝無用な生成・的外れな参照取得。無駄だが本丸は壊さない。
- よって**迷ったら固有/生成寄り**に倒すのを既定とする（変容性も得られ、本丸とも整合）。曖昧時の挙動は設定可能（§6.4）。

### 2.3 「正しい対象を取り違えない」精度が命
- 固有再現の品質は、生成エンジンの巧拙より**参照同定の正確さ**で決まる。"tower" で別の塔が来たら、後段が完璧でも全部無駄。
- だから v6 は v4 の「画像を1枚取る」を、**固有名 → 正規化クエリ → 候補取得 → vision検証で対象一致を確認 → 最良の1枚**という**識別**へ格上げする。

## 3. 実行環境
- v0〜v5 から基本変更なし。個人・ローカル利用前提（v4 §3）も継続（固有名＝実在物・版権キャラを扱うため、§R6）。
- 外部依存：画像検索（候補取得）＋ AI vision（一致検証＝Claude等）＋ 3D生成（v4）。複数依存はアダプタ隔離し、どれが落ちても全体を落とさない（§8）。

## 4. アーキテクチャ上の位置づけ
- v6は **v5ディスパッチャの「新規建築（new）」枝を賢くする**層。v5の編集（recolor/rescale/move等）・キャッシュ・削除のロジックは**不変**。
- v6が足すのは「新規建築の**生成元選択**」と「**リファレンス識別**」のみ。生成（v4）・パラメトリック（v1/v2）・ボクセル化（v3）・build/Undo（v2.x/v0）はすべて既存を呼ぶだけ。
- 出口は IR（パラメトリック）または GridIR（生成）。どちらも `build(ir, origin)` に収束。**seam不変**。

## 5. スコープ
### 5.1 v6でやること（In Scope）
- **分類器**：新規建築発話を 固有／ジェネリック／曖昧 に分類（AI言語、信頼度付き）。
- **ディスパッチャ**：ジェネリック→パラメトリック（v2型選択）、固有→リファレンス識別→生成（v4）、曖昧→既定ポリシー。
- **リファレンス識別**：固有名の正規化クエリ構築 → 候補画像取得 → **AI visionで対象一致・単体・クリーン・正面寄りを検証/再ランク** → 最良1枚。
- **信頼度＆フォールバック**：低信頼/同定不能時の明示通知・確認・降格（§6.5）。
- v5キャッシュとの統合（固有でも既知subjectはライブラリ決定論ロード優先）。

### 5.2 v6でやらないこと（Out of Scope：明示）
- **house/tower 等パラメトリック生成器の多様化・魔改造**（改訂原則6：固有には原理的に届かないため投資しない）。
- **3D生成エンジン自体の改良**（v4の領分。v6は「正しい参照を渡す」までが責務）。
- **複数対象の合成・シーン構築**（v5同様、単一対象）。
- **下流（build/Undo/buildGrid/voxelize）の再実装**。不変。
- **AIによる形・座標・voxel生成**。一切なし（生成はv4 stage3のみ）。

### 5.3 将来拡張（設計余地）
- 分類カテゴリの細分化（固有・ジェネリック・スタイル付きジェネリック等）。
- リファレンスを画像でなく**実在3Dモデル探索**へ拡張（忠実度優先に振るなら v3経路を固有に接続。現状は「喋るだけ」優先で画像→生成）。
- 参照の複数アングル取得で生成品質向上（v4側の強化と連携）。

## 6. コンポーネント詳細
### 6.1 分類器（AI言語）
- 入力：新規建築の発話（例「東京タワー作って」「塔作って」「ヨーロッパ風の家」）。
- 出力：`{ category: "specific" | "generic" | "ambiguous", subject: string, styleHint?: string, confidence: number }`。
- 分類指針：
  - **specific（固有）**：固有名詞・特定の実在物・名前のあるキャラ（「東京タワー」「スカイツリー」「自由の女神」「特定キャラ名」）。
  - **generic（ジェネリック）**：一般名詞・型・スタイル付き一般名（「塔」「家」「壁」「ヨーロッパ風の家」「適当な小屋」）。スタイル形容は generic のまま（styleHintへ）。
  - **ambiguous（曖昧）**：固有とも一般とも取れる（「お城」「教会」等、特定を指すか不明）。
- **AIは分類のみ**。形・座標に触れない。曖昧/低信頼は次段ポリシーへ。

### 6.2 ディスパッチャ（決定論）
分類結果に応じて生成元を選ぶ：
- **generic**：
  - パラメトリックの**型が存在する**（house/tower/wall/platform/box にマップできる）→ **v2の型選択フロー**へ（styleHintはpaletteへ）。生成を呼ばない。
  - 対応する型が**無い**（例「城」「橋」など未実装タイプ）→ ジェネリックでも**生成（v4）にフォールバック**（汎用生成）。
- **specific**：**リファレンス識別（§6.3）→ 生成（v4）**。ただし既知subjectは先に**v5ライブラリを確認**し、ヒットなら決定論ロード（生成を呼ばない）。
- **ambiguous**：**既定ポリシー（§6.4）**に従う。
- ディスパッチ判断自体はコードの決定論。AIは分類を返すだけ。

### 6.3 リファレンス識別（固有経路の中核）
固有名から**正しい参照**を取り違えずに得る。v4 stage2（画像取得）を本コンポーネントに**格上げ**する。
1. **正規化クエリ構築**（AI言語）：subjectを、対象を一意に絞る検索語へ正規化（例：曖昧さ回避の補語付加、正式名称化、ランドマーク/キャラとしての限定）。
2. **候補取得**（外部）：画像検索で**複数候補**を取得（1枚で決め打ちしない）。
3. **vision検証/再ランク**（AI vision）：各候補について「(a) 対象が指定の固有物と一致するか、(b) 単体被写体か、(c) 背景がクリーンか、(d) 正面寄りか」を評価し、スコアリング。**最良の1枚**を選ぶ。
   - これは画像の**一致確認・選別**であり、形には触れない（v3 §6.5の「色割当」と同じく、AIの判定が許される範囲）。
4. 選定画像を v4 生成（stage3以降）へ渡す。フォールバック用に保持（v4踏襲）。
5. 確信できる候補が無い → **同定不能**として §6.5 へ。

### 6.4 曖昧時のポリシー（設定可能）
- 既定：**生成寄りに倒す**（§2.2の非対称性＋変容性の観点。固有を箱に潰すより、生成して多様な結果を返す方が本丸と整合）。
- 任意：**確認モード**（曖昧時に「特定の○○ですか？一般的な○○ですか？」と1問だけ確認）。「喋るだけ」の摩擦を最小化したい場合は既定（無確認・生成寄り）、取り違えを嫌う場合は確認モード。
- ポリシーは設定値。低信頼（confidence < 閾値）も曖昧として扱う。

### 6.5 信頼度＆フォールバック（無言失敗禁止）
- **分類低信頼** → 曖昧ポリシー（§6.4）。
- **リファレンス同定不能**（一致候補なし/全候補低スコア）→ 無言で変なものを建てない。以下のいずれか（設定可）：
  - ユーザーに通知＋確認（「○○の画像を特定できませんでした。別の言い方/画像提供できますか？」）、
  - ジェネリック降格（対応する型があれば parametric で代替）、
  - 取得できた最良候補で v4 を試行し、破綻すれば v4 既存の平面フォールバックへ。
- いずれも**理由を通知**する（原則10）。

### 6.6 build連携
- generic経路は パラメトリックIR、specific経路は GridIR を産み、いずれも `build(ir, origin)` へ。**run-merge・facing・体積分割・Undo は不変**。
- v5の編集/キャッシュ/削除は v6 の上でそのまま機能（建てた後の「もっと大きく」等は v5 が捌く）。

## 8. 非機能要件（v6差分）
- v0〜v5 の非機能要件は継続。
- ロギングに **分類結果・信頼度・選んだ経路・正規化クエリ・候補スコア一覧・フォールバック有無**を追加。固有再現は「なぜその参照を選んだか」を追えることが重要。
- 外部依存（画像検索／vision／3D生成）を**アダプタ隔離**。失敗の切り分けと差し替えを容易に。鍵は環境変数。
- 分類器・正規化・vision検証は単体テスト（既知の固有/ジェネリック例、紛らわしい同名対象で誤分類・取り違えを検出）。
- 誤分類の非対称性（§2.2）に沿って、閾値は「固有→ジェネリックの誤りを優先的に避ける」方向で初期設定。

## 11. 既知のリスク・要確認事項（v6）
| # | 項目 | 内容・対処 |
|---|---|---|
| R1 | 誤分類（固有↔ジェネリック） | 非対称（固有→ジェネリックが致命）。閾値は固有寄りに初期化、曖昧は生成寄り。ログで調整（§2.2・FR-96）。 |
| R2 | 参照の取り違え | 同名異物（例：地名かぶり、同名キャラ）。vision検証で一致確認＋複数候補から選別（§6.3）。それでも限界はある。 |

## 付録A：ルーター擬似コード（v5ディスパッチャの new 枝を差し替え／実装は index.ts の routeNewBuild）
```ts
// v5: interpret() が EditOp を返す。kind==="new" の処理を v6 ルーターへ。
async function handleNewBuild(subject_utterance: string, origin: Vec3) {
  const c = await classify(subject_utterance);          // ① 固有/ジェネリック/曖昧（AI）
  // 既知subjectはまずキャッシュ（v5）
  const cached = library.find(c.subject);
  if (cached) return placeAsCurrent(library.load(cached), origin);
  const route = decideRoute(c, policy);                 // ② 決定論ディスパッチ（曖昧はpolicy）
  if (route === "parametric") {
    const ir = await selectTypeAndParams(c);            // v2: 型選択＋穴埋め（styleHint→palette）
    return placeAsCurrent(ir, origin);                  // パラメトリックIR → build
  } else { // "generation"
    const ref = await identifyReference(c.subject);     // ③ 正規化→候補→vision検証→最良1枚
    if (!ref) return handleUnidentified(c, origin);     // 同定不能：通知/確認/降格（§6.5）
    const ir = await generateViaV4(ref);                // 生成（v4, ref画像から）→ GridIR
    return placeAsCurrent(ir, origin);                  // GridIR → build
  }
}
```
形を作るのは `generateViaV4`（生成）だけ。`classify`/`identifyReference` はAIの言語・vision**判定**であり、`decideRoute`・`build` は決定論。v5の編集・キャッシュ・削除は一切変更しない。

## 付録B：ロードマップ上の位置づけ
```
v0   箱（seam確立）
v1   house（パラメトリック）
v2   タイプ複数化（生成器ライブラリ）
v2.x grid（器・密版）
v3   ボクセル化（充填）
v4   喋るだけ（生成+清掃+保険）
v5   ライブラリ＋修正ループ（GridIRが通貨）
v6   経路ルーター＋リファレンス識別  ← 固有/ジェネリックを1入口で振り分け、固有は正しい参照で生成
```
- v6で **「東京タワーとスカイツリーが別物として建つ」** が実現し、改訂後の本丸（固有の対象を喋るだけで作り分ける）に到達する。
- `build(ir, origin)` seam は v0からv6まで不変。v6も seam の上流（新規建築の生成元選択）に層を足しただけ。
- 固有再現の品質を握るのは生成エンジンより**参照同定の精度**である、という改訂方針が、v6の中核（§6.3）として実装される。
