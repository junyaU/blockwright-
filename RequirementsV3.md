## 0. このドキュメントの読み方（Claude Codeへの前提指示）

- 本書は **v0〜v2.x への差分**。既存の方針・制約はすべて継続有効。特に以下を**絶対に壊さない**：
  - **`build(ir, origin)` の署名は不変**。v3は `build()` より**上流**にステージを足すだけ。下流（`buildGrid`・ワールド変換・分割・送信・Undo）は一切変えない。
  - **IRは絶対座標を持たない**。v3の出力は v2.x の **GridIR**（ローカル空間データ）。
  - **AIは幾何を作らない**（v3はこれをさらに徹底＝§2）。
- **v3の核心：grid の"充填手段"を決定論的に作る。** v2.xで「器（GridIR）」を定義した。v3は「その器を正確に埋める」。埋め方は**ボクセル化（決定論）**であり、AI/生成モデルに形を作らせない。
- **道Aと道Bの切り分けを厳守**（§2.2）。v3の対象は**道A（リファレンスからの決定論ボクセル化）**。**道B（1枚絵からの3D復元）は v3 のスコープ外**（§5.2）。
- スコープを広げない。特に「text→3D生成」「単視点depth推定」は v3 では**やらない**（§9 R1）。

## 1. 背景と目的

v2.x で grid という器はできたが、当面の充填はテストフィクスチャ手書きのみだった。v3 のゴールは、**リファレンス（画像・3Dモデル）から GridIR を決定論的に生成するエンジン**を作り、「ロボットを建てて」のような**自由形状キャラ建築**を実用域に乗せること。

これまでの整理の総決算：自由形状は「誰が grid を埋めるか」で品質が決まる。**リファレンスがあるなら決定論ボクセル化が正解**で、AIは形に触れず**素材マッピング**だけを担う。立体ロボットは「確定的なリファレンスがある自由形状」なので、まさにこの道に乗る。

## 2. 設計思想

### 2.1 幾何は決定論、AIは素材マッピングのみ

```
リファレンス ──[ボクセル化：決定論]──▶ 占有(どのセルを置くか) + 色 ──[量子化]──▶ GridIR
                                                                  ↑
                                          AIの関与はここだけ（任意）：色/領域→ブロックIDの割当
```

- **占有（形）の決定は100%コード**。AI/LLMは voxels の占有を一切決めない。
- **色→ブロックの割当**は、既定で**決定論の最近傍色マッチ**（§6.4）。AIは「より良いブロック選び」の**任意の補助**に留まる（例：ピンク領域を pink_wool 系に寄せる）。AI無しでも完成する。
- これにより「LLMは3D座標が苦手」を完全に回避する。AIの仕事は v2.x からさらに縮退し、**形には絶対に触れない**。

### 2.2 道Aと道B（厳守）

| 道 | 入力 | 方式 | v3での扱い |
|---|---|---|---|
| **道A** | 3Dモデル(.obj/.glb) ／ 平面ドット絵 | 決定論ボクセル化（リファレンス有り） | **v3の対象** |
| 道B | 立体イラスト1枚 | 単視点3D復元（depth推定） | **スコープ外**（§5.2・R1） |

- 道Bは奥行きが原理的に欠落し、推定（でっち上げ）が要る＝精度が出ず破綻する。**v3では作らない。** 将来やるなら独立した実験トラックとして切り出す。

### 2.3 段階デリバリ（プラミング先行）

v3は2段で出す。**易しい入力で「画像→GridIR→build」の配管を先に通し**、その後に本命の3Dへ。

- **v3.0：平面ドット絵画像 → GridIR**（最易・ほぼAI不要）。配管検証。
- **v3.1：3Dモデル(.obj/.glb) → GridIR**（本命・立体キャラ）。道Aのボクセル化エンジン。

これは本プロジェクト一貫の「動かしながら育てる／易しい経路で配管を先に通す」方針に従う。

## 5. スコープ

### 5.1 v3でやること（In Scope）

- **v3.0**：平面画像（PNG等）→ GridIR。アルファ透過→air、画素色→ブロック量子化、厚み(thickness)パラメータ。
- **v3.1**：3Dモデル(.obj/.glb)→ GridIR。スケール合わせ、占有（shell/solid）、表面色サンプリング、色量子化。
- **色→ブロック量子化テーブル**（buildable block の代表色テーブル＋最近傍マッチ）。
- 量子化結果から **realized palette**（実際に使うブロック集合）を構築して GridIR に格納。
- **任意**：AIによる素材マッピング補助（色クラスタ/領域→ブロックID提案）。既定はコード決定論。
- ファイル入力経路（開発トリガー）。
- 既存基盤（GridIR・buildGrid・ワールド変換・体積分割・Undo）の流用。

### 5.2 v3でやらないこと（Out of Scope：明示）

- **道B：1枚絵からの3D復元／単視点depth推定**。
- **text→3D / 画像→3D の生成モデル利用**（AIに形を作らせる系すべて）。
- buildGrid 以降の再実装（不変）。
- アニメーション・可動・複数モデル合成。
- スラブ/階段/向き依存ブロックでの精密表現（v3は full solid block のみ、§6.4）。
- テクスチャの完全再現（あくまで色量子化レベル）。

### 5.3 将来拡張（設計余地）

- 道B（単視点復元）を独立実験トラックとして将来追加可能に（入口を差し替えるだけ、出口はGridIR）。
- ブロックテーブルを差し替え/拡張可能に（向き依存ブロック対応、map-art系の色拡張）。
- AI素材マッピングを段階的に強化（領域セグメンテーション連携など）。

## 6. コンポーネント詳細

### 6.1 共通：出力契約

- v3の各エンジンの**唯一の出口は GridIR**（v2.x §6.2準拠）。`size`・`voxels[y][z][x]`・`palette`。
- **次元順序・軸は v2.x に厳密準拠**（`voxels[y][z][x]`、y上方向、正面 lz=0）。入力側の軸系との変換（フリップ）を正しく行う（§6.2/6.3、R2）。
- realized palette：量子化で実際に使われたブロックのみを index 化（0=air予約）。

### 6.2 v3.0：平面画像 → GridIR

入力：PNG等（アルファ付き可）、目標サイズ（幅 or 高さ）、`thickness`（奥行き、既定1）。

手順：
1. 画像読込（画素RGBA取得）。
2. 目標サイズへ**area-average縮小**（1画素=1ブロックにするための間引き。色は領域平均）。
3. 各画素 (px, py)：alpha < 閾値 → air（skip）。それ以外 → 色量子化（§6.4）でブロック決定。
4. **軸マッピング**（重要）：`x = px`、`y = (rows-1) - py`（**画像は上が原点なので上下反転**）、`z = 0..thickness-1`（厚み分コピー）。
5. GridIR を構築（size, voxels[y][z][x], realized palette）。

> 2Dドット絵はこの段で「ほぼAI不要・決定論で正確」。配管（画像→grid→build→Undo）の検証に使う。

### 6.3 v3.1：3Dモデル → GridIR（道A本命）

入力：.obj / .glb（.stlも可）、目標解像度（例 高さ H ブロック）、`fill: "shell" | "solid"`（既定 solid）。

手順：
1. **メッシュ読込**：三角形（頂点座標・法線・UV・マテリアル/テクスチャ or 頂点色）。
2. **スケール合わせ**：メッシュAABBを求め、目標寸法（H）に合うvoxelサイズを算出。アスペクト比保持で w/h/d を導出し、**v2.xのサイズ上限（各≤64）にクランプ**。
3. **占有判定**：
   - `shell`：各三角形をvoxelグリッドにラスタライズ（三角形-voxel交差）→ 通過セルを占有（中空）。
   - `solid`：shell後、グリッド境界から外側をflood-fill → 「外側でも殻でもない」セルを内部として占有（中身詰まり）。小型キャラは既定solidで頑丈に。
4. **表面色サンプリング**：占有セルのうち表面セルは、最寄り三角形の色（UV→テクセル / マテリアル基本色 / 頂点色）を取得。内部セル（solid）は最寄り表面色 or 既定色。
5. **色量子化**（§6.4）：各セル色 → ブロックID。
6. **軸マッピング**：メッシュ座標系（glTFはY-up＝Minecraftと一致）を local(x,y,z) に対応付け。ハンドネス/前面方向のフリップを正しく行う（R2）。正面を lz=0 に合わせる。
7. GridIR を構築。

### 6.4 色 → ブロック量子化（決定論コア）

- **buildable block 代表色テーブル**を事前計算で持つ（`block_id → 代表RGB`）。
- **対象ブロックの制約**（テーブルに入れてよいブロック）：
  - full・opaque・solid のみ（スラブ/階段/薄板/透過は不可）。
  - **重力ブロック禁止**（sand/gravel等＝落ちる）。
  - redstone系・発光で紛らわしいもの・側面依存テクスチャの強いものは避ける。
  - 推奨コア：**concrete / wool / terracotta**（色網羅が良く面が均一）。木材・石材系で補完。
- **最近傍マッチ**：サンプル色を **Lab色空間**に変換して最近傍ブロックを選ぶ（RGB距離より知覚的に良い、R6）。
- realized palette はマッチで実際に選ばれたブロックのみで構成。

### 6.5 AI素材マッピング（任意・形には触れない）

- 既定は §6.4 の決定論で完結。**AIは任意の上乗せ**。
- AIに渡してよいのは「色クラスタ一覧」「ファイル名等の意味ヒント（例 robot→体はピンク）」程度で、**voxelの占有・座標は渡さない・触らせない**。
- AIの出力は「色/領域 → ブロックID の対応表（palette提案）」のみ。提案は §6.4 と同じ**素材検証**を通す。AI不在でもパイプラインは完成する。

### 6.6 build連携・Undo

- 生成 GridIR を `build(ir, origin)`（= `buildGrid`）に渡すだけ。**run-merge・facing・体積分割・全体AABB Undo は v2.x のまま**。v3側で再実装しない。
- 解像度が高いとブロック/コマンドが増える。v2.xの run-merge とサイズ上限に依存（R5）。

## 8. 非機能要件（v3差分）

- v0〜v2.x の非機能要件は継続。
- ロギングに **入力ファイル・目標解像度・占有voxel数・realized palette・量子化フォールバック・生成fill数**を追加。
- ボクセル化（占有・色サンプリング・量子化）は**単体テスト**を用意。特に**軸マッピング**は非対称フィクスチャ（左右・上下・前後が判別できる形）で転置/鏡像化を検出（R2）。
- 大きなモデルは時間がかかりうるので、ボクセル化と送信を分離（先に GridIR を作って検証 → build）できる構造にする。

## 10. 技術スタック（推奨）

- 画像：`sharp` または `jimp`/`pngjs`（画素取得・縮小）。
- メッシュ：.obj は自前パース可。.glb/.gltf は gltf ローダ（three.js 等のジオメトリ読込）。ボクセル化は三角形ラスタライズ＋flood-fillを自前 or 既存voxelizer。
- 色：Lab変換は小さなユーティリティ or color系ライブラリ。
- ブロック代表色テーブルは**事前生成のJSON**として同梱（オフラインで一度作る）。
- ファイル構成例：
  ```
  src/voxelize/
    image.ts      # v3.0: 画像→GridIR
    mesh.ts       # v3.1: メッシュ→GridIR（占有・色サンプリング）
    quantize.ts   # 色→ブロック（Lab最近傍）
    blocks.json   # buildable block 代表色テーブル
    palette.ts    # AI素材マッピング（任意）
  ```

## 付録A：パイプライン擬似コード（道A・メッシュ）

```ts
function voxelizeMesh(file: string, targetHeight: number, fill: "shell"|"solid"): GridIR {
  const mesh = loadMesh(file);                     // 三角形・UV・色
  const { w, h, d, voxelSize, aabb } = fitToGrid(mesh, targetHeight); // ≤64クランプ
  const occ = occupancy(mesh, w, h, d, voxelSize, aabb, fill);        // 決定論：形
  const palette: Record<number,string> = { /* 0:air予約 */ };
  const voxels = make3D(w, h, d, 0);               // [y][z][x]、既定0=air
  for (const cell of occ) {                        // 占有セルのみ
    const rgb = sampleSurfaceColor(mesh, cell);    // 表面色
    const block = quantizeLab(rgb);                // §6.4 最近傍（検証済みテーブル）
    const idx = internIndex(palette, block);       // realized palette へ
    const [x,y,z] = toLocalAxes(cell);             // 軸変換（R2）
    voxels[y][z][x] = idx;
  }
  return { type: "grid", size: { w, h, d }, voxels, palette };
}

// 接続：build以降は不変
const ir = voxelizeMesh("assets/robot.glb", 20, "solid");
const result = build(ir, origin);                  // = buildGrid（v2.x）
```

AIを使う場合も、関与は `quantize`/`palette` 段の**色割当のみ**。`occupancy`（形）には決して触れない。
