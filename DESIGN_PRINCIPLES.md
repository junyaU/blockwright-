# 設計方針：Minecraft統合版 AI建築システム

> このドキュメントは、本プロジェクト全体（v0〜v5）を貫く**設計思想**を明文化したもの。
> 個別の要件定義書（`requirements_..._v0〜v5.md`）の**上位**に位置する北極星であり、
> 新しいバージョンや機能を足すときの判断基準として参照する。

---

## 0. 一言で言うと

> **正しい境界で関心を分離し、各層を独立に育てられるようにする。**

このプロジェクトの全設計判断は、一本の境界 `build(ir, origin)`（IR seam）を中心に回っている。
AIの「賢さ」とコードの「正確さ」をこの境界で分け、**境界より上流に層を足していくだけ**で、
「箱が建つ」から「喋るだけで対象を作って・貯めて・直せる」まで積み上げてきた。
境界そのものは v0 から v5 まで**一度も変えていない**。これが本プロジェクトの背骨である。

**本丸（プロジェクトの狙い）**：**喋るだけで、固有の対象（特定の実在建造物・特定のキャラクター等）を、データ用意ゼロで建てる**こと。
忠実さより手軽さを優先する。この狙いが、以下の原則（特に原則6）の優先順位を決めている。
「ジェネリックな家や塔を画一的に量産すること」は狙いではなく、固有の対象を**作り分けられる**ことが価値の中心である。

## 1. 核となる原則

### 原則1：唯一の継ぎ目を固定する

`build(ir, origin)` の**署名と契約は不変**。すべての機能は、この境界の**上流に層を足す**形で実装する。
下流（座標計算・コマンド生成・送信・Undo）は再実装しない。

- **なぜ**：retrofit（後からの作り直し）が最も高くつく継ぎ目はここ一点だから。ここさえ固定すれば、他はすべて「あとから安く足せる」。
- **現れ方**：v1〜v5 はすべて seam の上流に層（生成器・ボクセル化・キャッシュ・修正ループ）を足しただけ。
- **禁じること**：機能追加のために build の署名や下流を書き換えること。

### 原則2：AIは設計、コードは施工

AIは**意図・パラメータ・構造化された中間表現（IR）**を作る。**座標と幾何は決定論コード**が作る。

- **なぜ**：LLMは精密な3D座標計算と逐次的な状態追跡が苦手。ここを任せると、壁ズレ・宙ぶらりん・左右非対称・回路の沈黙が出る。
- **現れ方**：house は AI がw/d/h/屋根を出し、buildHouse が座標を組む。voxel の占有はコードが決め、AIは色割当だけ。
- **禁じること**：AIに生 `setblock`／座標／voxel占有を出させること。

### 原則3：IRは絶対座標を持たない

IRは「**何を**建てるか」だけを表す。「**どこに**建てるか」は `origin` として外から与える。

- **なぜ**：同じIRを任意の場所に建てられる。配置と形状が直交する。
- **配当**：v5で「移動」が `origin` を変えるだけで済む。形を一切触らずに位置だけ動かせる。

### 原則4：決定論を既定にし、AIの関与を最小化・局所化する

AIが触れてよいのは**最小の安全な面**だけ。版を追うごとにAIの守備範囲を**意図的に縮めてきた**。

| 版 | AIの仕事 |
|---|---|
| v1 | パラメータを埋める |
| v2 | タイプを選ぶ＋パラメータを埋める |
| v3 | 素材マッピング（色→ブロック）だけ。形に触れない |
| v4 | （唯一の例外）stage3で形を生成。それ以外は言語・取得・決定論 |
| v5 | 修正発話の言語分類＋palette割当だけ |

- **なぜ**：不確実性の源（AI/生成）を絞るほど、システム全体の堅牢さが上がる。
- **禁じること**：「便利だから」とAIに決定論で済む判断を渡すこと。

### 原則5：器を先に、充填は後で

ターゲット表現（器）を**正しく定義してから**、それを埋める手段（充填）を足す。器は将来の充填に耐える形で設計する。

- **なぜ**：器を間違えると、充填手段を作る段で器ごと作り直しになる。
- **現れ方**：v2.x で grid IR を**密版**で定義（疎は密で表せるが逆は不可）。v3 のボクセル化は、その器に充填エンジンを差すだけで済んだ。
- **判断軸**：「この器は、まだ見ぬ充填手段に耐えるか？」を先に問う。

### 原則6：対象の性質で経路を選ぶ（固有の対象は生成が本道）

対象が**ジェネリック**（型で表せる・画一的でよい）なら**決定論の専用生成器**。対象が**固有**（特定の実在物・唯一のシルエットを持つ）なら**生成が本道**。

- **なぜ**：パラメトリックは「同じ金型のパラメータ違い」しか出せない。東京タワーとスカイツリーのように**シルエットの構造そのものが違う固有の対象**は、パラメータの軸に個性が存在しないため原理的に作り分けられない。固有性は固有のリファレンスからしか出てこない。
- **本丸との関係**：本プロジェクトの本丸は「固有の対象を喋るだけで建てる」こと。したがって**生成（v4）が本道**であり、パラメトリック（house/tower）は「ジェネリックな対象」専用の脇役に降格する。**house/tower の多様化（部品増し・味付け）には投資しない**（固有の作り分けには原理的に届かないため）。
- **判断軸**：「これは固有の対象か、ジェネリックか？」固有なら生成、ジェネリックでよいならパラメトリック。
- **生成は本道でも非決定論のまま**：本道に格上げしても荒さは消えない。原則7（隔離）とフォールバックで囲う前提は維持する。本道だから無防備に使う、ではない。
- **旧方針からの転換**：以前は『パラメトリックを優先し、生成は最後の逃げ道』としていた。これは『正確さ・画一性で足りる』前提に依存していた方針である。本丸が固有の対象の再現である以上、その前提は固有の対象には当てはまらない。よって原則6は『常に成り立つ法則』から『**ジェネリックな対象に限って成り立つ法則**』へ格下げした（改訂メモ参照）。

### 原則7：不確実性は隔離する

非決定論（生成）を導入せざるを得ないときは、**1段に封じ込め、前後で吸収し、保険を置く**。不確実性をシステム全体に伝播させない。

- **なぜ**：荒さを放置すると下流すべてが汚染される。
- **現れ方（v4）**：形の生成は stage3 だけ。荒さは stage4（cleanup）で吸収。stage5以降は決定論。破綻したら平面建築にフォールバック。
- **合言葉**：荒さは生まれた場所で囲い込み、決定論の領域には漏らさない。

### 原則8：動かしながら育てる（ただし継ぎ目だけ先に）

スコープはミニマム先行。上から完璧に設計しない。**ただし retrofit が高くつく境界だけは v0 から引く**。配管は**易しい入力で先に通す**。

- **なぜ**：AI×外部連携×非公式APIの不確実性は、上から読み切れない。動かして学ぶのが速い。
- **現れ方**：v0は箱だけ建てる骨格。だがIR seamは最初から引いた。v3は平面画像（最易）で配管を通してから3Dへ。
- **判断軸**：「これは後から安く足せるか、それとも継ぎ目か？」継ぎ目だけ先に、残りは育てる。

### 原則9：安いことに金を払わない

決定論で済むことを、AIにも生成器にも渡さない。**安い操作と高い操作に段差をつける**。

- **現れ方（v5）**：色替え・大きさ・移動・反転は**キャッシュ済みGridIRの決定論変形**。形を変えるときだけ生成器に戻る。
- **判断軸**：「これは既存データの決定論変形で済むか？」済むなら生成を呼ばない。

### 原則10：黙って失敗しない／前提を正直に書く

検証ゲートで破綻を検知し、失敗は**明示通知**する。非公式API・ライセンス・受け入れたトレードオフは**要件に明文化**する。

- **現れ方**：無言失敗禁止。WSプロトコルは「実装前に疎通実験で確認」。v4は「ローカル個人利用限定」「原則を破る版」と明記。フォールバック時は理由を通知。
- **なぜ**：隠れた前提と沈黙する失敗が、最も追跡困難なバグを生む。

## 2. 関心の分離（AI / コードの境界）

| 領域 | 担当 | 性質 |
|---|---|---|
| 意図抽出・タイプ選択・修正分類 | AI | 言語 |
| パラメータ充填・素材（色→ブロック）割当 | AI | 言語/意味（形に触れない） |
| 形の生成（v4） | AI/生成器 | 固有の対象では**本道**。ただし唯一の非決定論＝隔離対象 |
| 座標計算・幾何生成・占有判定 | コード | 決定論 |
| ボクセル化・色量子化・変形（recolor/rescale等） | コード | 決定論 |
| 配置・送信・Undo | コード | 決定論 |

**AIの守備範囲は「言語」と「色割当」に縮退していくのが正しい方向**。形・座標・占有は、生成という1つの島を除き、常にコード側にある。生成は固有の対象にとっては本道だが、非決定論である点は変わらないので、原則7（隔離）とフォールバックで囲い続ける。

---

## 3. 判断ヒューリスティクス（新機能を設計するとき）

1. **固有の対象か、ジェネリックか？** 固有（特定の実在物・唯一のシルエット）なら生成が本道。ジェネリック（型で表せる・画一でよい）なら専用生成器（決定論）。
2. **決定論で済むか？** 既存データの変形・計算で済むなら、AI/生成器を呼ばない。
3. **これは継ぎ目か、後付けか？** retrofitが高い境界だけ先に固める。残りはミニマムで動かしてから育てる。
4. **不確実性をどこに閉じ込めるか？** 非決定論を入れるなら1段に隔離し、前後で吸収し、保険を置く。
5. **器は将来の充填に耐えるか？** 表現（IR/データ構造）は、まだ見ぬ拡張に耐える形で先に正しく定義する。

## 4. アンチパターン（やらないこと）

- AIに生 `setblock`／座標／voxel占有を出させる（原則2違反＝中継実装、後から全書き直し）。
- **固有の対象（特定の実在物・キャラ）をパラメトリックで建てる**（原則6違反＝同じ金型しか出ず、固有の作り分けという本丸を殺す）。
- ジェネリックで十分な対象（街に置くだけの家・塔）を生成に流す（無用な非決定論＝原則6/7の反対側の失敗）。
- house/tower を「固有の対象も作れるように」多様化・魔改造する（原理的に届かない＝労力の浪費。固有は生成へ）。
- 器を疎で定義してから密が必要になる（原則5違反＝器ごと作り直し）。
- 生成の荒さを cleanup/隔離せず下流に流す（原則7違反＝決定論領域の汚染）。
- 決定論で済む修正のために生成器を呼ぶ（原則9違反＝安い操作に高い金）。
- build の署名・下流を機能追加のために書き換える（原則1違反＝seamの崩壊）。
- 無言で失敗する／前提を暗黙にする（原則10違反）。

## 5. ロードマップが原則を体現する形

```
v0   箱（ループ＋seam）            原則1（seam確立）・原則8（動かす土台）
v1   house（パラメトリック）         原則2（設計/施工）・原則6（ジェネリックはパラメトリック）
v2   タイプ複数化（生成器ライブラリ） 原則4（AI=選択+穴埋め）・原則2
v2.x grid（自由形状の器・密版）       原則5（器を先に）
v3   ボクセル化（充填手段）           原則5（充填は後で）・原則4（AIは色だけ）
v4   喋るだけ（生成+清掃+保険）       原則7（不確実性の隔離）★唯一の原則破りを封じ込め
v5   ライブラリ＋修正ループ           原則9（安い/高いの段差）・原則1（GridIRが合流点）
```

全バージョンは `build(ir, origin)` seam の**上流に層を足しただけ**。核は一度も変わっていない。

## 6. 新しいバージョンを足すときのチェックリスト

新機能・新バージョンを設計する前に、以下を満たすか確認する：

- [ ] `build(ir, origin)` の署名を変えていないか（原則1）
- [ ] 下流（buildGrid以降・配置・Undo）を再実装していないか（原則1）
- [ ] AIに座標・幾何・voxel占有を出させていないか（原則2・4）
- [ ] IRに絶対座標を持ち込んでいないか（原則3）
- [ ] 対象の性質（固有／ジェネリック）に応じた経路を選んでいるか。固有の対象をパラメトリックで潰していないか（原則6）
- [ ] 新しい表現（器）は将来の拡張に耐える形か（原則5）
- [ ] 非決定論を入れるなら1段に隔離し、吸収層と保険があるか（原則7）
- [ ] 決定論で済むことを生成器/AIに渡していないか（原則9）
- [ ] retrofitが高い継ぎ目だけ先に引き、残りはミニマムにしているか（原則8）
- [ ] 失敗は明示通知で、前提（非公式API・ライセンス・トレードオフ）は明文化したか（原則10）

このチェックリストを通れば、新バージョンは**既存の8枚の要件・本方針と整合**したまま、seam の上に積み増せる。

## 7. 最後に：このプロジェクトが守ってきたもの

このプロジェクトの価値は「Minecraftに何を建てられるか」よりも、**「AIと決定論コードをどう分業させるか」の一貫した解答**にある。

- AIは**苦手なこと（精密な幾何）から遠ざけ**、得意なこと（言語・意図・選択・色の意味づけ）に集中させる。
- 不確実性は**避けるのではなく、隔離して飼い慣らす**（v4の原則破りすら、封じ込めとフォールバックで囲った）。
- そのすべてを、**一本の境界 `build(ir, origin)` の上に積む**ことで、各層が独立して育てられる状態を保つ。

「喋るだけで固有の対象を建てる」という派手なゴールの裏で、実際に効いていたのはこの地味な規律だった。
新しい機能で迷ったら、**派手さではなくこの規律に従う**——それが本プロジェクトの設計方針である。

## 改訂メモ

- **原則6の転換**：当初は「パラメトリックを優先し、生成は最後の逃げ道」だった。だが本丸を「**固有の対象（特定の実在建造物・キャラ）を喋るだけで建てる**」と定めた時点で、この優先順位は固有の対象には当てはまらなくなった。パラメトリックは「同じ金型のバリエーション」しか出せず、東京タワーとスカイツリーのような**シルエットが構造的に違う対象を作り分けられない**ためである。よって原則6を「対象の性質で経路を選ぶ（固有の対象は生成が本道）」へ改め、パラメトリック（house/tower）は**ジェネリックな対象専用の脇役**に降格、その多様化には投資しない方針とした。
- **本丸の入力前提**：「あくまで喋るだけ・データ用意ゼロ」を優先する。よって主戦場は v4（image→3D生成）の強化であり、忠実さで勝る v3（実在3Dモデルのボクセル化）は、設備（モデル探索）が増えるため主役には据えない。
- **次にやるべきこと**：固有とジェネリックを判定して経路を振り分ける**経路ルーター**と、固有名から正しい参照画像を取る**リファレンス識別の精度**。生成エンジンの精度より、この「正しい対象を取り違えない」精度の方が、固有再現には効く。
